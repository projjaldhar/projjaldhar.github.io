Overview

Podcast content is rich with insights, but fundamentally unsearchable — hours of audio locked away with no way to query specific topics or moments. This project tackles that problem by building a production-grade Retrieval-Augmented Generation (RAG) system that transforms podcast episodes into a conversational knowledge base.

The system handles the full pipeline autonomously: downloading episodes, transcribing audio with Deepgram, chunking and embedding the transcripts, storing vectors in Pinecone, and surfacing answers through a Streamlit interface — all without manual intervention at any stage.

The Problem

Unsearchable audio

Valuable podcast knowledge is trapped in audio form — impossible to query, skim, or reference without listening in full.

Keyword search limitations

Traditional keyword search fails to capture meaning — missing semantically related content and returning irrelevant exact-match results.

Manual processing overhead

Manually transcribing, organising, and tagging episodes to build any kind of knowledge base is prohibitively time-consuming at scale.

Solution & Architecture

The system implements a complete RAG pipeline: audio is downloaded and sent to Deepgram for transcription, the resulting text is chunked into overlapping segments and converted into dense vector embeddings, which are indexed in Pinecone. At query time, the user's question is embedded using the same model, the nearest vectors are retrieved from Pinecone, and a language model generates a grounded answer from those retrieved chunks.

Key Features

  • Automated episode management — downloads and processes new podcast episodes without manual intervention, keeping the knowledge base current
  • Natural language queries — users ask questions in plain English; semantic search retrieves the most contextually relevant transcript segments
  • Grounded, contextual answers — responses are generated directly from retrieved podcast content, reducing hallucination and anchoring answers in the source material
  • Interactive Streamlit dashboard — displays indexed episodes, supports conversational follow-up questions, and shows source citations for transparency
  • Scalable vector store — Pinecone handles similarity search at scale with low-latency retrieval, supporting growing episode libraries without re-architecture

Learning & Outcomes

Building this system end-to-end surfaced the real tradeoffs in production RAG: chunk size and overlap strategy significantly affect retrieval quality, embedding model selection involves cost-vs-accuracy tradeoffs, and prompt engineering for grounded generation requires careful iteration to prevent the model from reaching beyond retrieved context.

The project was completed as part of the Curious PM March 2026 cohort, demonstrating hands-on AI product building — from architecture decisions and infrastructure setup to user interface design and iterative quality improvement. The result is a working system that turns passive podcast listening into an active, queryable knowledge base.

Lessons Learned

  • Chunk size and overlap strategy has outsized impact on answer quality — smaller chunks improved retrieval precision but stripped the surrounding context that the LLM needed to generate coherent answers. Finding the right balance required more experimentation than expected and should be treated as a tunable parameter from the start.
  • Transcription quality is a hidden upstream dependency — Deepgram performed well generally, but technical terminology and proper nouns were occasionally transcribed incorrectly, degrading downstream retrieval. A transcript cleaning step before embedding would have improved results without significant effort.
  • RAG prompt engineering is its own discipline — standard prompting techniques didn't prevent the model from occasionally reaching beyond the retrieved context. Explicit, tested instructions anchoring the model strictly to provided chunks were necessary to make answers trustworthy.
  • Embedding costs are easy to underestimate at scale — embedding every word of every episode without a smart chunking and batching strategy was expensive. Designing for cost-efficiency from the start — batching requests and avoiding redundant re-embedding — is worth the upfront planning.
