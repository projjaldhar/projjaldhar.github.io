Overview

Kohls operated one of the largest department store networks in the US, with inventory spread across hundreds of stores, distribution centres, and e-commerce fulfilment nodes. Without a unified inventory visibility layer, stock decisions were being made on stale or incomplete data — leading to overselling, missed fulfilment opportunities, and inefficient stock transfers between locations.

This project implemented IBM Sterling's Global Inventory Visibility model to provide Kohls with a real-time, enterprise-wide view of inventory — enabling accurate availability calculations, smarter fulfilment routing, and eliminating the blind spots in their existing inventory management infrastructure.

The Problem

Siloed inventory systems

Store, DC, and e-commerce inventory lived in separate systems with no unified view, making enterprise-wide stock decisions unreliable.

Stale availability data

Batch-updated inventory feeds created windows of inaccuracy that led to overselling online and missed ship-from-store opportunities.

Inefficient fulfilment routing

Without real-time cross-node visibility, orders were frequently routed to suboptimal fulfilment locations, increasing cost and delivery time.

Solution & Architecture

IBM Sterling's GIV model was implemented to aggregate inventory data across all nodes — stores, DCs, and online fulfilment centres — into a single, real-time visibility layer. The implementation included interface configuration to connect existing inventory systems, API integration for real-time stock queries, and the configuration layer that enabled availability calculations to draw from the unified data store.

Key Features

  • Unified inventory view — real-time aggregation of stock levels across all Kohls nodes into a single queryable data layer
  • Multi-system interface integration — configured Sterling GIV interfaces to receive inventory updates from existing store, DC, and e-commerce systems without requiring those systems to be replaced
  • Real-time availability API — Java API layer exposing live inventory availability for downstream availability calculations and fulfilment routing decisions
  • Availability reservation logic — soft reservation capability to prevent overselling during the window between availability check and order confirmation
  • Full design documentation — High Level Design and Detailed Level Design documents produced, providing Kohls with a durable architectural record of the implementation

Impact & Outcomes

The GIV implementation gave Kohls a real-time, enterprise-wide inventory visibility layer that had not previously existed. This provided the data foundation for more accurate availability promises to customers online, smarter fulfilment routing decisions, and the elimination of the overselling incidents that had been occurring due to stale stock data.

Leading a team of five developers through this implementation was my first experience in a technical lead role, giving me a foundation in stakeholder management, design documentation, and delivery accountability that I carried directly into product management.

Lessons Learned

  • Data quality from source systems is the foundation everything else depends on — GIV was only as accurate as the inventory feeds coming into it. Investing time upfront in validating source system data quality before building on it prevented downstream accuracy issues.
  • Incremental validation beats big-bang testing — validating inventory accuracy node by node before aggregating gave the team confidence and caught interface issues early, rather than discovering them during a full-scale integration test.
  • Leading a team requires explicit communication, not assumption — my first time managing five developers taught me that what's obvious to the lead is often unclear to the team. Daily stand-ups and written design summaries made alignment visible rather than assumed.
  • The business problem is always more important than technical elegance — early in the project I spent time on technically sophisticated interface patterns that the client neither needed nor valued. Learning to solve the business problem with the minimum necessary complexity was the most lasting lesson from this engagement.
