Overview

Williams Sonoma operated a complex direct-to-consumer order management landscape across multiple legacy systems — PKMS for warehouse management, RMS for retail management, and DTCMO for direct-to-consumer operations. These systems operated in silos, creating friction at integration points and limiting the agility of the DTC fulfilment operation.

The goal was to migrate the entire DTC order flow to IBM Sterling as a unified platform — replacing the legacy systems incrementally while keeping live operations running. This was a high-stakes implementation for a premium retail brand where order fulfilment accuracy and customer experience were non-negotiable.

The Problem

Fragmented legacy systems

PKMS, RMS, and DTCMO each handled parts of the DTC order lifecycle but required manual intervention at handoff points, creating operational overhead and error risk.

Integration brittleness

Interfaces between legacy systems were custom-built and fragile — changes to one system frequently caused failures in connected systems, increasing maintenance cost and risk.

Limited fulfilment flexibility

The existing architecture couldn't easily support new fulfilment models — the rigidity of the legacy systems blocked the business from implementing capabilities like ship-from-store.

Solution & Architecture

The migration was executed as a phased Sterling implementation — progressively moving DTC order flows from the legacy systems onto Sterling while maintaining parallel operation to protect live transactions. Custom interfaces and APIs were developed to bridge Sterling with existing systems during the transition, with each phase retiring another legacy component as Sterling capabilities went live.

Key Features

  • Unified DTC order management — single Sterling platform replacing three legacy systems, providing a consistent order lifecycle view across the entire DTC operation
  • Legacy system bridge interfaces — custom-developed interfaces maintaining integration with PKMS, RMS, and DTCMO during the parallel-run transition period without disrupting live operations
  • Custom API development — bespoke Sterling Java APIs configured to meet Williams Sonoma's specific DTC order flow requirements beyond out-of-the-box functionality
  • Phased rollout with rollback safety — each migration phase designed with rollback capability, ensuring that live DTC operations could revert to the legacy system if validation failed
  • Agile production support model — daily monitoring, incident management, and enhancement activities maintained throughout the migration to ensure zero degradation in customer-facing order fulfilment

Impact & Outcomes

The phased migration successfully moved Williams Sonoma's DTC order flow onto IBM Sterling without disrupting live customer fulfilment operations. The replacement of three fragmented legacy systems with a unified Sterling platform reduced the integration maintenance burden and provided the architectural foundation for future DTC capabilities.

Managing this implementation — maintaining agile production support while executing a complex migration in parallel — required careful coordination between the client's operations team, the development team, and the Sterling platform capabilities. It was a formative project in understanding how to manage complex technical transitions without sacrificing operational stability.

Lessons Learned

  • Parallel running is the safest migration strategy — running Sterling alongside the legacy systems before cutting over, rather than doing a hard cutover, prevented the catastrophic failure modes that a big-bang migration would have risked for a live retail operation.
  • Undocumented legacy behaviour is the hidden risk in every migration — legacy systems accumulate undocumented business rules over years of patching. Discovering these mid-migration caused more delays than any technical challenge. Shadow-running early in the process to expose these gaps would have helped significantly.
  • The transition interfaces are temporary but must be treated as production quality — the bridge APIs connecting Sterling to legacy systems during parallel running were scoped as temporary, but they ran live operations. Cutting corners on their quality or testing would have put the migration at risk.
  • Agile ceremonies maintain rhythm during complex migrations — the daily standup and sprint cadence were what kept the team aligned when the scope and technical landscape were shifting constantly. Process discipline was the anchor in an otherwise ambiguous environment.
